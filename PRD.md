# PRD — OfficeBridge

> **OfficeBridge** — 클라이언트리스 ZTNA 게이트웨이
> 버전: v0.3 (2026-07-09) · 해커톤 24시간 MVP 기준
> 상태: 개발 진행 중 — F-1/F-2(임시)/F-3/F-7 구현·검증 완료

---

## 1. 한 줄 정의

**VPN 없이, 브라우저만으로, 인가된 사내 시스템에만 접근하게 하는 제로트러스트 게이트웨이.**
AI 정책 엔진이 모든 접속의 위험도를 실시간 평가한다.

## 2. 문제 정의

### 기존 방식 (VPN)의 문제

| 문제 | 설명 |
|---|---|
| 과도한 접근 권한 | VPN 접속 = 사내망 **전체** 진입. 계정 하나 탈취되면 내부 이동(lateral movement)으로 전산망 전체가 위험 |
| 공격 표면 노출 | VPN 장비가 인터넷에 직접 노출. 최근 수년간 대형 침해사고의 단골 진입점 (Ivanti, Fortinet 등 VPN 취약점) |
| 정적 신뢰 | 로그인 한 번 성공하면 세션 내내 신뢰. 세션 탈취·내부자 이상행위에 무방비 |
| 운영 부담 | 전 직원 PC에 클라이언트 설치·업데이트·장애 대응 |
| 느린 권한 회수 | 퇴사자·사고 계정의 접근 차단이 즉각적이지 않음 |

### 왜 지금인가

- 재택·외근·협력사 접근이 일상화 — 경계(perimeter) 기반 보안 모델 붕괴
- 글로벌은 ZTNA(Cloudflare Access, Zscaler, Tailscale)로 이동 중이나, **국내는 데이터 주권·컴플라이언스 이슈로 해외 SaaS 도입 장벽** → 국내 IDC 기반 서비스의 기회

## 3. 목표 / 비목표

### 해커톤 목표 (24h)

1. 외부 사용자가 **브라우저만으로** SSO 인증 후 사내 웹 시스템에 접속하는 E2E 데모
2. 정책 기반 접근 제어 (허용 앱 / 차단 앱)
3. **AI 위험도 평가** — 접속 컨텍스트를 LLM이 평가, 고위험 시 차단
4. 관리자 대시보드 — 실시간 접속 로그 + **원클릭 즉시 차단**
5. 발표자료 완성

### 비목표 (명시적으로 안 만드는 것)

- PC 에이전트 (비웹 프로토콜 TCP/UDP 터널링) → 로드맵 2단계
- 기기 신뢰도 검증 (백신/OS 패치 상태) → 로드맵 2단계
- 고가용성, 성능 최적화, 커넥터 이중화
- 자체 계정 체계 (SSO는 Google OAuth로 대체)
- 실제 IDC 배포 (데모는 로컬 또는 클라우드 VM 1대)

## 4. 타깃 고객 & 페르소나

- **1차 타깃:** 임직원 50~1,000명 규모, 웹 기반 전산시스템(그룹웨어·ERP·위키) 중심의 국내 중견기업. 보안팀이 있으나 소수 정예
- **관리자 페르소나 "보안팀 박 과장":** VPN 장비 CVE 대응에 지침. 퇴사자 계정 회수 누락이 감사 지적사항. 도입이 쉽고 눈에 보이는 통제를 원함
- **사용자 페르소나 "영업팀 김 대리":** 외근 중 노트북·태블릿으로 그룹웨어 접속. VPN 앱이 자꾸 끊겨서 스트레스

## 5. 제품 구성 (아키텍처)

```
[외부 사용자 브라우저] ──443──▶ [릴레이 서버 (개발서버 10.52.249.249:443)]
                                 ├ 게이트웨이 프록시      ✅ server.js
                                 ├ 인증 모듈             ✅ auth.js (임시 데모 로그인, OAuth 교체 예정)
                                 ├ 정책 엔진             ✅ policy.js + policy.json (정적 룰)
                                 │                       ⬜ AI 위험도 평가 (F-4)
                                 ├ 터널 매니저            ✅ server.js
                                 ├ 감사 로그             ⬜ (현재 콘솔 로그만)
                                 └ 관리자 대시보드        ⬜ (F-6)
                                      ▲
                                      │ 아웃바운드 WSS 터널 (인바운드 개방 0)
                             [커넥터 ✅ connector/index.js (고객사 사내망, 1대)]
                                      │
                             [사내 웹 시스템 ✅ mock-apps (그룹웨어:8081 위키:8082 인사DB:8083)]
                              — 127.0.0.1 전용 바인딩: 터널을 통해서만 접근 가능
```

- 사용자 PC: 설치물 없음 (클라이언트리스)
- 커넥터: 고객사당 1대, 사전 발급 토큰으로 릴레이에 등록 (멀티테넌트)
- 서비스 판별: 접속 호스트명의 첫 라벨 (`groupware.10.52.249.249.sslip.io` → `groupware`)
- 포트: **443 고정** (사내망 방화벽이 443만 인바운드 허용함을 실측 확인)

### 5.1 통신 흐름 상세 (구현 기준)

**[흐름 0] 터널 준비 — 커넥터 부팅 시 1회**

```
커넥터                                      릴레이
  │ ① ws://relay:443/tunnel?token=xxx ────▶ │  토큰 검증 (불일치 시 401 거부)
  │ ② {type:"register",                     │
  │    services:["groupware","wiki","hrdb"]}▶  서비스명→터널 매핑 등록
  │ ◀────────── 터널 유지 (30초마다 ping) ──▶ │
  │    끊기면 커넥터가 3초 후 자동 재연결        │
```

**[흐름 1] 미인증 사용자 → 로그인**

```
브라우저                                    릴레이
  │ ① GET groupware.~/ ──────────────────▶ │ 세션 쿠키 없음
  │ ◀── ② 302 /_ob/login?next=/ ────────── │
  │ ③ GET /_ob/login ────────────────────▶ │
  │ ◀── ④ 로그인 화면 (데모 계정 목록) ────── │
  │ ⑤ POST /_ob/login (계정 선택) ────────▶ │ 세션 생성 (메모리 Map)
  │ ◀── ⑥ Set-Cookie: ob_session=… ─────── │ 쿠키는 부모 도메인에 설정
  │        + 302 원래 주소                   │ → 모든 서비스 호스트에서 공유 (SSO 효과)
```

**[흐름 2] 인증된 사용자 → 허용된 앱 (예: 김영업 → 그룹웨어)**

```
브라우저          릴레이                          커넥터           사내 앱
  │ GET / ──────▶ │ ① 세션 확인 ✓                  │               │
  │  (쿠키 포함)   │ ② 정책: kim.sales에             │               │
  │               │    groupware 있음 → ALLOW      │               │
  │               │ ③ 요청을 JSON 봉투로 포장:       │               │
  │               │  {type:"request", id, method,  │               │
  │               │   path, headers, body(base64)} │               │
  │               │ ──── WSS 터널로 전송 ──────────▶ │               │
  │               │                                │ ④ services.json│
  │               │                                │  groupware →   │
  │               │                                │  127.0.0.1:8081│
  │               │                                │ ─ HTTP 요청 ──▶ │
  │               │                                │ ◀── HTML 응답 ─ │
  │               │ ◀─ {type:"response", id,       │               │
  │               │    status, headers, body} ──── │               │
  │ ◀─ HTTP 응답 ─ │ ⑤ id로 대기 중이던 요청 매칭     │               │
```

**[흐름 3] 인증된 사용자 → 미허용 앱 (예: 김영업 → 인사DB)**

```
브라우저                릴레이
  │ GET hrdb.~/ ──────▶ │ ① 세션 확인 ✓
  │                     │ ② 정책: kim.sales에 hrdb 없음 → DENY
  │ ◀── 403 차단 화면 ── │ ③ 요청이 터널로 가지 않음
                            = 사내망에 패킷이 도달조차 하지 않음 (제로트러스트 핵심)
```

> 판정 순서 요약: **`/_ob/*` 내부경로? → 인증(세션) → 정책(허용 앱) → 터널 전달**
> 실패 지점마다 각각 로그인 리다이렉트 / 403 / 503(커넥터 없음) / 504(사내앱 무응답)로 응답

## 6. 기능 요구사항

### P0 — 데모에 반드시 필요

| ID | 기능 | 상세 | 상태 |
|---|---|---|---|
| F-1 | 터널링 | 커넥터가 릴레이로 아웃바운드 WSS 연결 수립·유지·자동 재연결. 릴레이는 HTTP 요청/응답을 터널 위로 중계 | ✅ **완료** (실환경 검증: 맥→개발서버 443→터널→목업) |
| F-2 | 인증 | 미인증 요청 → 로그인 → 세션 쿠키 발급. **현재 데모 계정 선택형으로 구현** (OAuth 콜백·TLS 의존성 회피). 시간 남으면 Google OAuth로 교체 | ✅ 임시 완료 |
| F-3 | 정적 정책 | `policy.json` 사용자↔앱 매핑. 미허용 앱 403 차단 화면. 파일 수정 시 1초 내 자동 반영 (재시작 불필요) | ✅ **완료** |
| F-4 | AI 위험도 평가 | 접속 컨텍스트(시간대, 접속 이력 대비 이상 여부, 요청 빈도)를 Claude API로 평가 → 0~100 점수. 임계값 초과 시 차단 + 사유 표시 | ⬜ |
| F-5 | 감사 로그 | 모든 요청 기록: 사용자, 시각, 대상 앱, 판정(허용/차단), 위험도 점수, 사유 | ⬜ (콘솔 ALLOW/DENY 로그만) |
| F-6 | 관리자 대시보드 | 실시간 로그 스트림(SSE), 활성 세션 목록, 사용자 **즉시 차단** 버튼(세션 폐기) | ⬜ **다음 작업** |
| F-7 | 사내 시스템 목업 | 그룹웨어·위키·인사DB 3종. 그럴듯한 UI (데모 무대장치) | ✅ 완료 |

**데모 계정 (relay/policy.json):** 김영업·이개발 = 그룹웨어+위키 / 박인사 = 전체(인사DB 포함)

### P1 — 시간 남으면

| ID | 기능 | 상세 |
|---|---|---|
| F-8 | 고위험 시 추가 인증 | 차단 대신 step-up 인증(이메일 OTP 등) 요구 |
| F-9 | AI 이상행위 요약 | 대시보드에서 "오늘의 이상 접속 요약"을 LLM이 자연어로 리포트 |
| F-10 | 커넥터 상태 모니터링 | 대시보드에 커넥터 연결 상태·지연시간 표시 |

## 7. 데모 시나리오 (발표 2분)

현재 구현으로 **①②는 지금 바로 시연 가능**, ③④는 F-4/F-6 개발 후 가능.

1. **[정상 접속]** ✅ 노트북에서 `groupware.10.52.249.249.sslip.io:443` 접속 → OfficeBridge 로그인 화면 → 김영업 선택 → 그룹웨어 화면. 이어서 `wiki.~` 접속 → 재로그인 없이 바로 열림(SSO). *"VPN 없이요. 설치한 것도 없습니다."*
2. **[정책 차단]** ✅ 김영업 상태로 `hrdb.~` 접속 → 🛡️ 403 차단 화면. *"권한 없는 시스템엔 요청이 사내망에 도달조차 하지 않습니다."* (보너스: policy.json 수정 → 1초 내 반영되어 접근 허용되는 장면 연출 가능)
3. **[AI 차단]** ⬜ 시간 조작으로 "새벽 3시 + 대량 요청" 상황 재현 → AI 위험도 87점 → 차단 + 대시보드 경고. *"룰에 없는 이상행위도 AI가 잡습니다."* (F-4 필요)
4. **[즉시 차단]** ⬜ 관리자가 대시보드에서 차단 버튼 클릭 → 사용자 화면 즉시 세션 종료. *"퇴사자 권한 회수, 1초입니다."* (F-6 필요)

## 9. 차별점 (심사 Q&A 방어 논리)

1. **AI 동적 정책 엔진** — 기존 ZTNA는 정적 룰. 우리는 접속 컨텍스트를 LLM이 평가해 위험도 기반으로 동적 판단
2. **국내 IDC 기반** — 트래픽이 해외 클라우드를 경유하지 않음. 데이터 주권·컴플라이언스 민감 고객(금융·공공·제조) 공략
3. **도입 마찰 최소** — 고객사 방화벽 변경 0 (아웃바운드 443만), 직원 PC 설치물 0

## 10. 기술 스택

| 영역 | 선택 (실제 구현) | 비고 |
|---|---|---|
| 릴레이 서버 | Node.js 내장 http + ws 패키지 | 외부 의존성 ws 하나뿐. 포트 443 |
| 커넥터 | Node.js 단일 스크립트 (ws + 내장 fetch) | services.json으로 노출 앱 설정 |
| 대시보드 | 미정 — 단일 HTML + SSE 예정 (React까지 불필요할 수 있음) | F-6에서 결정 |
| 목업 앱 | Node.js 내장 http, 의존성 0 | 127.0.0.1 전용 바인딩 |
| AI | Claude API | F-4에서 사용 |
| DB | 현재 메모리(Map) + policy.json 파일 | 로그 영속화 필요 시 SQLite 검토 |
| 인증 | 데모 계정 로그인 (자체 세션 쿠키) | Google OAuth는 P1로 강등 |
