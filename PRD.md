# PRD — OfficeBridge

> **OfficeBridge** — 클라이언트리스 ZTNA 게이트웨이
> 버전: v0.8 (2026-07-10) · 해커톤 24시간 MVP 기준
> 상태: **기능·UI 개발 완료 (F-1~F-7 + 다크모드)** · 제출 레포(개인 public) 이관 완료 — 남은 것: 발표자료, 리허설, (선택) TLS

---

## 1. 한 줄 정의

**VPN 없이, 브라우저만으로, 인가된 사내 시스템에만 접근하게 하는 제로트러스트 게이트웨이.**
로그인 이후에도 매 요청을 재검증하는 행위 기반 지속 검증(Continuous Verification)으로 접근을 통제한다.

### 산출물 설명 (제출용, 약 240자)

> OfficeBridge는 VPN을 대체하는 클라이언트리스 제로트러스트 게이트웨이입니다. 직원은 브라우저만으로 인가된 사내 시스템(ERP·그룹웨어 등)에 접속하고, 커넥터가 아웃바운드 터널로 연결해 방화벽 인바운드 개방이 필요 없습니다. 부서·개인별 접근 정책, 로그인 후에도 매 요청을 재검증하는 행위 기반 제어(Continuous Verification), 실시간 감사 로그와 계정 즉시 차단을 관리자 콘솔에서 통합 관리합니다.

## 2. 문제 정의

### 기존 방식 (VPN)의 문제

| 문제 | 설명 |
|---|---|
| 과도한 접근 권한 | VPN 접속 = 사내망 **전체** 진입. 계정 하나 탈취되면 내부 이동(lateral movement)으로 전산망 전체가 위험 |
| 공격 표면 노출 | VPN 장비가 인터넷에 직접 노출. 최근 수년간 대형 침해사고의 단골 진입점 (Ivanti, Fortinet 등 VPN 취약점) |
| 정적 신뢰 | 로그인 한 번 성공하면 세션 내내 신뢰. 세션 탈취·내부자 이상행위에 무방비 |
| 운영 부담 | 전 직원 PC에 클라이언트 설치·업데이트·장애 대응 |
| 느린 권한 회수 | 퇴사자·사고 계정의 접근 차단이 즉각적이지 않음 |

### 왜 지금인가

- 재택·외근·협력사 접근이 일상화 — 경계(perimeter) 기반 보안 모델 붕괴
- 글로벌은 ZTNA(Cloudflare Access, Zscaler, Tailscale)로 이동 중이나, **국내는 데이터 주권·컴플라이언스 이슈로 해외 SaaS 도입 장벽** → 국내 IDC 기반 서비스의 기회

## 3. 목표 / 비목표

### 해커톤 목표 (24h)

1. 외부 사용자가 **브라우저만으로** SSO 인증 후 사내 웹 시스템에 접속하는 E2E 데모
2. 정책 기반 접근 제어 (허용 앱 / 차단 앱)
3. **AI 위험도 평가** — 접속 컨텍스트를 LLM이 평가, 고위험 시 차단
4. 관리자 대시보드 — 실시간 접속 로그 + **원클릭 즉시 차단**
5. 발표자료 완성

### 비목표 (명시적으로 안 만드는 것)

- PC 에이전트 (비웹 프로토콜 TCP/UDP 터널링) → 로드맵 2단계
- 기기 신뢰도 검증 (백신/OS 패치 상태) → 로드맵 2단계
- 고가용성, 성능 최적화, 커넥터 이중화
- 자체 계정 체계 (SSO는 Google OAuth로 대체)
- 실제 IDC 배포 (데모는 로컬 또는 클라우드 VM 1대)

## 4. 타깃 고객 & 페르소나

- **1차 타깃:** 임직원 50~1,000명 규모, 웹 기반 전산시스템(그룹웨어·ERP·위키) 중심의 국내 중견기업. 보안팀이 있으나 소수 정예
- **관리자 페르소나 "보안팀 박 과장":** VPN 장비 CVE 대응에 지침. 퇴사자 계정 회수 누락이 감사 지적사항. 도입이 쉽고 눈에 보이는 통제를 원함
- **사용자 페르소나 "영업팀 김 대리":** 외근 중 노트북·태블릿으로 그룹웨어 접속. VPN 앱이 자꾸 끊겨서 스트레스

## 5. 제품 구성 (아키텍처)

```
[외부 사용자 브라우저] ──443──▶ [릴레이 서버 (개발서버 10.52.249.249:443)]
                                 ├ 게이트웨이 프록시      ✅ server.js (인증→정책→터널 4단계 검문)
                                 ├ 인증 모듈             ✅ auth.js (이메일+비밀번호 SHA-256, 계정 차단)
                                 ├ 정책 엔진             ✅ policy.js — 부서(조직도) 정책 ∪ 개인 추가 권한
                                 ├ 행위 기반 지속 검증     ✅ rules.js — 벤더 룰셋 매 요청 재검증(F-4)
                                 ├ 터널 매니저            ✅ server.js (매 요청에 목적지 지시 → 커넥터 무설정)
                                 ├ 감사 로그             ✅ audit.js (메모리 1000건 + audit.log + SSE 구독)
                                 ├ 임직원 포털            ✅ portal.js — 전체 시스템 표시, 무권한 🔒 잠금
                                 └ 관리자 콘솔            ✅ admin.js — 사이드바 + 서브 URL 7페이지
                                      ▲
                                      │ 아웃바운드 WSS 터널 (인바운드 개방 0)
                             [커넥터 ✅ connector/index.js (고객사 사내망, 1대, 무설정)]
                                      │
                             [고객사(AMD) 사내 시스템 ✅ mock-apps]
                              ERP:8081 · GitLab:8082 · 그룹웨어:8083 · 회계관리:8084(미등록, 시연용)
                              — 127.0.0.1 전용 바인딩: 터널을 통해서만 접근 가능
```

- 사용자 PC: 설치물 없음 (클라이언트리스), 대표 진입점은 포털 `portal.~`
- 커넥터: 고객사당 1대, 사전 발급 토큰으로 릴레이에 등록 (멀티테넌트). 앱→내부주소 매핑은 릴레이가 관리
- 서비스 판별: 접속 호스트명의 첫 라벨 (`erp.10.52.249.249.sslip.io` → `erp`)
- 포트: **443 고정** (사내망 방화벽이 443만 인바운드 허용함을 실측 확인)
- 테넌트 표시: 전 화면에 고객사(AMD) 브랜딩, 디자인 토큰 통일 (라이트/네이비)

**관리자 콘솔 메뉴 (서브 URL):** 대시보드 `/dashboard` · 정책 설정(사내시스템 등록 `/policy/apps`, 정책 접근관리 `/policy/access`, 행위 기반 제어 `/policy/rules`, 활성세션·계정통제 `/policy/sessions`) · 로그 조회(시스템 접속로그 `/logs/system` — 허용/차단 필터, 관리자 로그 `/logs/admin`)

### 5.1 통신 흐름 상세 (구현 기준)

**[흐름 0] 터널 준비 — 커넥터 부팅 시 1회**

```
커넥터                                      릴레이
  │ ① ws://relay:443/tunnel?token=xxx ────▶ │  토큰 검증 (불일치 시 401 거부)
  │ ② {type:"register",                     │
  │    services:["groupware","wiki","hrdb"]}▶  서비스명→터널 매핑 등록
  │ ◀────────── 터널 유지 (30초마다 ping) ──▶ │
  │    끊기면 커넥터가 3초 후 자동 재연결        │
```

**[흐름 1] 미인증 사용자 → 로그인**

```
브라우저                                    릴레이
  │ ① GET groupware.~/ ──────────────────▶ │ 세션 쿠키 없음
  │ ◀── ② 302 /_ob/login?next=/ ────────── │
  │ ③ GET /_ob/login ────────────────────▶ │
  │ ◀── ④ 로그인 화면 (이메일+비밀번호 폼) ── │
  │ ⑤ POST /_ob/login ───────────────────▶ │ SHA-256 해시 검증
  │    (실패 시 401 + 에러 표시, 감사 기록)    │ 성공 시 세션 생성 (메모리 Map)
  │ ◀── ⑥ Set-Cookie: ob_session=… ─────── │ 쿠키는 부모 도메인에 설정
  │        + 302 원래 주소                   │ → 모든 서비스 호스트에서 공유 (SSO 효과)
```

**[흐름 2] 인증된 사용자 → 허용된 앱 (예: 김영업 → 그룹웨어)**

```
브라우저          릴레이                          커넥터           사내 앱
  │ GET / ──────▶ │ ① 세션 확인 ✓                  │               │
  │  (쿠키 포함)   │ ② 정책: kim.sales에             │               │
  │               │    groupware 있음 → ALLOW      │               │
  │               │ ③ 요청을 JSON 봉투로 포장:       │               │
  │               │  {type:"request", id, method,  │               │
  │               │   path, headers, body(base64)} │               │
  │               │ ──── WSS 터널로 전송 ──────────▶ │               │
  │               │                                │ ④ services.json│
  │               │                                │  groupware →   │
  │               │                                │  127.0.0.1:8081│
  │               │                                │ ─ HTTP 요청 ──▶ │
  │               │                                │ ◀── HTML 응답 ─ │
  │               │ ◀─ {type:"response", id,       │               │
  │               │    status, headers, body} ──── │               │
  │ ◀─ HTTP 응답 ─ │ ⑤ id로 대기 중이던 요청 매칭     │               │
```

**[흐름 3] 인증된 사용자 → 미허용 앱 (예: 김영업 → 인사DB)**

```
브라우저                릴레이
  │ GET hrdb.~/ ──────▶ │ ① 세션 확인 ✓
  │                     │ ② 정책: kim.sales에 hrdb 없음 → DENY
  │ ◀── 403 차단 화면 ── │ ③ 요청이 터널로 가지 않음
                            = 사내망에 패킷이 도달조차 하지 않음 (제로트러스트 핵심)
```

> 판정 순서 요약: **`/_ob/*` 내부경로? → 인증(세션) → 정책(허용 앱) → 터널 전달**
> 실패 지점마다 각각 로그인 리다이렉트 / 403 / 503(커넥터 없음) / 504(사내앱 무응답)로 응답
> **모든 판정은 감사 로그에 기록** (LOGIN OK/FAIL, ACCESS ALLOW/DENY, LOGOUT, SYSTEM 장애 — 누가·언제·어디서(IP)·무엇을·왜)

## 6. 기능 요구사항

### P0 — 데모에 반드시 필요

| ID | 기능 | 상세 | 상태 |
|---|---|---|---|
| F-1 | 터널링 | 커넥터가 릴레이로 아웃바운드 WSS 연결 수립·유지·자동 재연결. 릴레이는 HTTP 요청/응답을 터널 위로 중계 | ✅ **완료** (실환경 검증: 맥→개발서버 443→터널→목업) |
| F-2 | 인증 | 이메일+비밀번호 로그인 (SHA-256 해시 검증, 실패 시 401+에러 표시+감사 기록). 로그인 화면에 데모 계정 자동입력 칩. 세션 쿠키는 서비스 호스트 간 공유(SSO). OTP·패스키·OIDC는 로드맵 | ✅ **완료** |
| F-3 | 정책 엔진 | **조직도(부서) 정책 + 개인 추가 권한** (최종 권한 = 부서 ∪ 개인). 관리자 콘솔에서 칩 토글로 부여/회수, 즉시 반영. 미허용 앱 403 차단 | ✅ **완료** (조직도 확장) |
| F-4 | 행위 기반 지속 검증 | **(방향 변경: AI 점수 → 결정적 룰셋)** 벤더 제공 룰셋을 매 요청 재검증(Continuous Verification). ① 자체 룰 5종: 업무시간 외 차단 / 과도한 요청(10초 30회) / 세션 IP 고정(탈취 방지, 세션 종료) / 로그인 연속 실패 잠금(10분 5회) / 세션 최대 수명(8h). ② **엔드포인트 연동(별도 카드)**: OfficeKeeper 등 DLP의 PC 보안 점수 80점 이상만 허용 — UI·토글만 구현, 판정 로직은 엔드포인트 연동 예정(`uiOnly`). 관리자는 ON/OFF만, 위반은 접속로그에 `[행위기반]` 차단 기록 | ✅ **완료** |
| F-5 | 감사 로그 | 구조화 이벤트 기록(id·시각·종류·판정·사용자·앱·IP·사유): 메모리 링버퍼 1,000건 + audit.log JSONL 영속화 + 실시간 구독(F-6용) + 조회 API `/_ob/api/logs?token=` | ✅ **완료** (실환경 검증) |
| F-6 | 관리자 콘솔 | 사이드바+서브URL 7페이지: 대시보드(타일+실시간 SSE 로그) / **사내시스템 등록**(라벨+내부주소 → 포털·라우팅 즉시 반영) / **정책 접근관리**(부서·개인) / **행위 기반 제어**(룰셋 ON/OFF) / **활성세션·계정통제**(세션 종료, 계정 차단·해제) / **시스템 접속로그**(허용·차단 등 필터 버튼) / **관리자 로그**(모든 관리자 조치 이력) | ✅ **완료** |

## 7. 데모 시나리오 (발표 2~3분)

**③(AI) 외 전부 지금 시연 가능.**

1. **[정상 접속]** ✅ 노트북에서 포털 `portal.10.52.249.249.sslip.io:443` 접속 → OfficeBridge 로그인 → 김영업(비밀번호 demo1234) → 포털에 ERP·그룹웨어는 열리고 GitLab은 🔒 잠금 → ERP 클릭 → K-System 로그인 화면 도달. *"VPN 없이요. 설치한 것도 없습니다."*
2. **[정책 차단]** ✅ 김영업 상태로 `gitlab.~` 직접 접속 → 🛡️ 403 차단. *"권한 없는 시스템엔 요청이 사내망에 도달조차 하지 않습니다."* → 관리자 콘솔 [정책 접근관리]에서 영업팀 또는 김영업 개인에 gitlab 부여 → 새로고침 → 즉시 열림. *"권한 부여, 1초."*
3. **[행위 기반 차단]** ✅ ⓐ [행위 기반 제어]에서 "업무시간 외 접속 차단" ON → 저녁 발표장에서 즉시 403. ⓑ 새로고침 연타(10초 30회) → 자동화 의심 차단. ⓒ 비밀번호 5회 실패 → 계정 자동 잠금 → 관리자가 해제. *"로그인은 시작일 뿐, 모든 요청을 계속 검증합니다."*
4. **[즉시 차단]** ✅ [활성세션·계정통제]에서 김영업 **계정 차단** → 사용자 새로고침 시 즉시 로그아웃 + 재로그인 거부("차단된 계정"). *"퇴사자 권한 회수, 1초입니다."*
5. **[온보딩]** ✅ (보너스) [사내시스템 등록]에서 회계관리(acct, http://127.0.0.1:8084) 라이브 등록 → 포털에 타일 즉시 생성 → 권한 부여 → 접속. *"시스템 온보딩, 1분입니다."*
6. **[직접 접근 차단 증명]** ✅ `10.52.249.249:8081` 직접 접속 → 타임아웃. *"이 시스템들은 인터넷에 존재하지 않습니다. 유일한 입구는 OfficeBridge뿐입니다."*

## 9. 차별점 (심사 Q&A 방어 논리)

1. **AI 동적 정책 엔진** — 기존 ZTNA는 정적 룰. 우리는 접속 컨텍스트를 LLM이 평가해 위험도 기반으로 동적 판단
2. **국내 IDC 기반** — 트래픽이 해외 클라우드를 경유하지 않음. 데이터 주권·컴플라이언스 민감 고객(금융·공공·제조) 공략
3. **도입 마찰 최소** — 고객사 방화벽 변경 0 (아웃바운드 443만), 직원 PC 설치물 0

## 10. 기술 스택

| 영역 | 선택 (실제 구현) | 비고 |
|---|---|---|
| 릴레이 서버 | Node.js 내장 http + ws 패키지 | 외부 의존성 ws 하나뿐. 포트 443 |
| 커넥터 | Node.js 단일 스크립트 (ws + 내장 fetch) | 무설정 실행 — 라우팅은 릴레이가 지시 |
| 관리자 콘솔/포털 | 서버 렌더 단일 HTML + SSE + fetch (프레임워크 없음) | admin.js / portal.js |
| 목업 앱 | Node.js 내장 http, 의존성 0 | 127.0.0.1 전용 바인딩 |
| 지속 검증 | 자체 룰(결정적) + 엔드포인트(OfficeKeeper) 연동 | AI(Claude API) 미채택 — 결정적 룰이 데모 안정성↑ |
| DB | 메모리(Map) + policy.json + audit.log(JSONL) | SQLite 불필요해짐 |
| 인증 | 이메일+비밀번호 (SHA-256, 자체 세션 쿠키) | OTP·패스키·OIDC는 로드맵 |
| 운영 | start.sh — 3개 프로세스 일괄 재시작 + 자가진단 | 개발서버: `git pull && ./start.sh` |
